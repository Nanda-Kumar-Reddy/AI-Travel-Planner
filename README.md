# AI Travel Planner

> Generate day-by-day travel itineraries with AI, get a real-time confidence score, and let the Risk Co-Pilot flag pacing problems, budget overruns, and weather risks — with one-click AI fixes.

**Live URLs:**
- **Frontend:** https://ai-travel-planner-xi-one.vercel.app/
- **Backend API:** https://ai-travel-planner-j6r3.onrender.com/health

> ⚠️ The backend runs on Render's free tier. If the health check returns slowly (20–40 seconds), the instance is cold-starting. Subsequent requests are fast.

---

## 1. Project Overview

AI Travel Planner is a full-stack web application that takes a destination, duration, budget tier, and interests as inputs, then produces a day-by-day itinerary, hotel recommendations, and a budget breakdown — all generated by Google Gemini 2.5 Flash in a single API call.

The differentiating layer is the **Trip Confidence & Risk Co-Pilot**: a three-algorithm engine that independently audits every itinerary for pacing problems (haversine distance between consecutive activities), budget overruns (actual vs. tier-expected spend), and weather risk (Open-Meteo live forecast or seasonal climatology). The result is a numeric confidence score (0–100) and a set of actionable risk flags. Clicking "Fix this" on any flag triggers a targeted Gemini regeneration that resolves the specific issue while preserving the rest of the trip.

The project was built across 9 phases over 5 days.

## 3. Setup Instructions

### Prerequisites

- Node.js 18+
- A MongoDB Atlas cluster (free tier works)
- A Google Gemini API key from [aistudio.google.com](https://aistudio.google.com/apikey)

### Backend

```bash
cd backend
cp .env.example .env
# Fill in: MONGO_URI, JWT_SECRET, GEMINI_API_KEY, FRONTEND_URL
npm install
npm run dev       # starts on http://localhost:5000
```

**`backend/.env.example` variables:**

| Variable | Description |
|---|---|
| `PORT` | Server port (default 5000) |
| `NODE_ENV` | `development` or `production` |
| `MONGO_URI` | MongoDB Atlas connection string |
| `JWT_SECRET` | 64-char random hex (`openssl rand -hex 32`) |
| `GEMINI_API_KEY` | Google Gemini API key |
| `FRONTEND_URL` | CORS origin (e.g., `http://localhost:3000`) |
| `WEATHER_MOCK` | `true` to return a deterministic weather flag (for demo — see Known Limitations) |

### Frontend

```bash
cd frontend
cp .env.example .env.local
# Set NEXT_PUBLIC_API_URL=http://localhost:5000

```

**`frontend/.env.example` variables:**

| Variable | Description |
|---|---|
| `NEXT_PUBLIC_API_URL` | Backend URL (`http://localhost:5000` in dev; Render URL in prod) |

### Deployed environment

For the live deployment, the backend `FRONTEND_URL` is set to the Vercel deployment URL, and `NEXT_PUBLIC_API_URL` in Vercel's environment variables points to the Render backend URL.

## 5. Auth & Authorization Approach

Authentication uses **JWT tokens stored in httpOnly cookies** — the token is invisible to JavaScript, which eliminates the token-theft-via-XSS attack vector that the more common `localStorage` approach leaves open.

**Session lifecycle:**
1. `POST /api/auth/register` or `POST /api/auth/login` → server issues JWT (7-day expiry), sets `Set-Cookie: token=...; HttpOnly; Secure; SameSite=None`
2. Every subsequent request from the browser automatically includes the cookie
3. The `requireAuth` middleware verifies the JWT and attaches `{ id, email }` to `req.user`
4. `POST /api/auth/logout` → server sets `maxAge: 0` on the cookie to clear it; client state is also cleared via Zustand

**The cross-origin fix (Phase 8):** The frontend (Vercel) and backend (Render) are different origins. Browsers silently drop cookies on cross-origin requests unless three conditions are met: `SameSite=None`, `Secure=true`, and `credentials: 'include'` on the fetch. All three are implemented. Without the `SameSite=None` change, login appeared to succeed but every subsequent request returned 401.

For the full auth design rationale, see [docs/BACKEND.md → Auth Design](./docs/BACKEND.md#3-auth-design).

## 6. AI Agent Design

The Gemini integration is structured as four independent generation functions in `backend/src/services/gemini.service.ts`:

| Function | Purpose | Zod schema |
|---|---|---|
| `generateItinerary()` | Full trip generation (itinerary + hotels + budget) | `TripGenerationResponseZ` |
| `regenerateDayActivities()` | Single-day regeneration (general or risk-fix) | `RegenerateDayResponseZ` |
| `generateBudgetEstimate()` | Budget refresh (pins activities cost) | `BudgetRefreshResponseZ` |
| `generateHotelSuggestions()` | Hotel refresh (exactly 3, one per tier) | `HotelsRefreshResponseZ` |

**Resilience pattern:**
1. All calls go through `withExponentialBackoff`: 5 retries with delays of **1s → 2s → 4s → 8s → 16s**
2. `responseMimeType: 'application/json'` activates Gemini's JSON mode — dramatically reduces markdown-wrapped or malformed output
3. Every response is validated with **Zod** before any database write. On validation failure: the prompt is amended with a validation-failure note and retried once. If the second attempt also fails Zod, an `AppError(502)` is thrown
4. Primary model: `gemini-2.5-flash-preview-05-20`. On model-unavailable errors, falls back to stable `gemini-2.5-flash`

For the complete integration details and prompt engineering, see [docs/BACKEND.md → Gemini Integration](./docs/BACKEND.md#4-gemini-integration).

## 7. Creative Feature: Trip Confidence & Risk Co-Pilot

Most AI travel planners stop at generation. This one audits its own output.

After a trip is generated, the Risk Co-Pilot runs three independent algorithms:

- **Pacing check** — haversine distance between consecutive same-day activities. Flags pairs more than 35 km apart as high-severity, 15–35 km as medium.
- **Budget check** — compares total estimated cost against tier-based expected spend ($100/day Low, $300/day Medium, $700/day High). Also flags per-day activity cost spikes.
- **Weather check** — calls Open-Meteo for a live precipitation forecast (trips within 16 days) or the previous year's archive data as a seasonal climatology proxy (trips further out or with no start date).

The result: a confidence score between 0 and 100, computed as `100 − (15 × high_flags) − (8 × medium_flags) − (3 × low_flags)`, and a set of specific, actionable risk flags with suggested fixes.

When a user clicks "Fix this" on a pacing flag, a targeted regeneration prompt instructs Gemini to replace that day's activities with venues clustered within 5 km of each other — resolving the structural issue, not just rephrasing the same activities. The confidence score visibly updates afterward (animated SVG ring + numeric counter).

This is the feature that closes the trust gap. Every other submission can produce a plausible itinerary. An itinerary planner that catches its own mistakes and fixes them live is doing something structurally different.

## 8. Key Design Decisions, Trade-offs, and Known Limitations

These are the actual deviations and fixes from this build's real history — not generic disclaimers.

### Design decisions

**Vite → Next.js (Phase 1 correction):** The initial scaffolding used Vite + React. This was replaced with Next.js 14 before any product code was written because: Vite is a bundler with a dev server, not a framework; it has no built-in routing, no `next/dynamic` for SSR bypass, no optimized font loading, and no deployment path to Vercel without additional configuration. Next.js solved all of these at once.

**localStorage → httpOnly cookies (Phase 2 decision):** The assessment reference used `Authorization: Bearer` with localStorage. This was rejected because any script on the page can read localStorage — including injected scripts from XSS. The httpOnly cookie pattern closes that attack surface at the cost of slightly more CORS configuration.

**Custom `ThemeProvider` instead of `next-themes`:** ~100 lines of React replaces a 3 KB dependency. The custom implementation matches `next-themes`' API surface, includes the same anti-flash blocking script, and handles OS theme changes via `matchMedia` event listeners.

### Known limitations

| Limitation | Detail |
|---|---|
| **SameSite cross-origin fix required** | Local dev works with `SameSite: Lax`. Cross-origin deployment (Vercel + Render) required `SameSite: None` + `Secure: true` + `credentials: 'include'`. This was discovered and fixed in Phase 8. |
| **LLM-estimated pricing** | Budget figures and hotel prices come from Gemini's training data, not live pricing APIs. Actual costs at time of travel will differ. |
| **Weather > 16 days = climatology, not forecast** | Trips more than 16 days out use last year's archive precipitation data as a seasonal proxy. Flags generated this way include "(seasonal estimate — not a live forecast)" in the message text. |
| **`WEATHER_MOCK=true`** | A deterministic fake weather flag for demo reliability. Disclosed in the flag text. Should be `false` in production. |
| **Pacing is straight-line distance** | The haversine check computes crow-fly distance, not routing time. A 30 km flag might be 25 minutes by metro or 2 hours in traffic. |
| **Render cold starts** | The backend on Render's free tier sleeps after 15 minutes. First request after sleep takes 20–40 seconds. |
| **No rate limiting on trip creation** | `POST /api/trips` triggers multiple Gemini calls. A burst of requests could exhaust the API quota. Not implemented within the 5-day build constraint. |
| **No offline support** | No service worker or local cache. The app requires network connectivity. |
| **Client-rendered protected pages** | Dashboard and trip detail pages fetch data on mount. No SSR for auth-gated content. |

For complete details, see [docs/BACKEND.md → Known Limitations](./docs/BACKEND.md#7-known-backend-limitations) and [docs/FRONTEND.md → Known Limitations](./docs/FRONTEND.md#7-known-frontend-limitations).
